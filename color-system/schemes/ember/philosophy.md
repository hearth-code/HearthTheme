# Ember Philosophy

Ember is a warm-neutral color philosophy for code interfaces and reading surfaces inside HearthCode.

It treats color as a semantic language before it becomes a platform theme.
The goal is not to make every product look identical.
The goal is to let users feel the same atmosphere, hierarchy, and emphasis priorities across VS Code, Obsidian, and the website.

## Core Thesis

HearthCode should feel:

- warm but not muddy
- expressive but not noisy
- readable for long sessions
- stable across dark, soft, and light climates

## Vocabulary

The palette speaks through a small set of named families:

- `ember`: control-flow pressure and release accents
- `mineral`: a bounded cool anchor for structure
- `sage`: calm literal rhythm
- `ochre`: dry structural classification
- `chalk`: readable carrier neutrals

## Role Philosophy

Colors are assigned by meaning before widget.

- `keyword` should be memorable but never flood the page
- `function` should anchor structure without making the system feel cold
- `string` should support reading rhythm
- `comment` should guide quietly
- surfaces should reduce glare before they express personality

## Variant Philosophy

The four variants share one hierarchy and one personality.
What changes is the climate:

- `dark`: warm default
- `darkSoft`: quieter night atmosphere
- `light`: paper-tone daylight
- `lightSoft`: calmer long-form reading light

Soft variants are not lower-contrast copies.
They are distinct climates with the same semantic contract.

## Product Principle

Ember is the warm-neutral HearthCode line, not a pile of platform tokens.
The framework can later host other schemes, but every scheme should still flow through the same five layers:

1. philosophy / scheme manifest
2. color language core
3. variant system
4. platform contracts + calibration
5. generated artifacts + lineage
